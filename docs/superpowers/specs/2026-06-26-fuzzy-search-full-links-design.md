# Fuzzy Search Full Links Design

## Goal

Replace the current fuzzy search candidate buttons with one full-detail link per visible candidate row.

When a user runs `/search <name>` or sends plain text for fuzzy search, the bot should return candidate rows that each include a clickable `Full` link. Tapping that link should directly return the full sanctions detail for the selected candidate, without requiring the user to copy a name, type another command, or choose from a separate button block below the result list.

This change applies only to fuzzy search candidate results. Existing exact lookup results may keep their current `/basic` and `/full` inline callback buttons.

## Existing Context

The project currently has:

- Fuzzy candidate search through `DebarmentService.searchCandidates`.
- Fuzzy result text rendering in `formatFuzzySearchResult`.
- Candidate action buttons generated by `candidateActionButtons`.
- Exact-result action buttons generated by `actionButtons`.
- Existing callback handling for `basic:<recordId>` and `full:<recordId>`.
- Existing record-id detail lookup through `fullByRecordId`.
- Telegram reply rendering in `replyToContext`, which currently sends text plus optional inline keyboard buttons.

The previous fuzzy search detail-button design added `/basic N` and `/full N` inline buttons under the search result list. This design replaces that fuzzy-search-specific button block with row-level full links.

## Telegram Interaction Constraint

Telegram message text links can open URLs, but they do not send hidden `callback_data` to the bot. Hidden callback data is available through inline keyboard buttons.

To support row-level clickable text while still triggering the bot, use Telegram bot deep links:

```text
https://t.me/<bot_username>?start=full_<recordId>
```

Telegram will deliver this to the bot as:

```text
/start full_<recordId>
```

The bot then parses the start payload, verifies access, and runs `fullByRecordId(recordId)`.

The deep-link payload must stay within Telegram's supported payload characters and length. Current OpenSanctions-style record ids are compatible with `A-Z`, `a-z`, `0-9`, `_`, and `-`, so the payload can be `full_<recordId>` without encoding for current data. If future record ids include unsupported characters or exceed the payload limit, the implementation should fall back to a short token mapping rather than exposing invalid links.

Official references:

- Telegram Bot Features, Deep Linking: `https://core.telegram.org/bots/features#deep-linking`
- Telegram Bot API, InlineKeyboardButton: `https://core.telegram.org/bots/api#inlinekeyboardbutton`

## User Flow

1. User sends `/search Yatai Smart` or plain text `Yatai Smart`.
2. Bot returns `Possible matches`.
3. Each candidate row includes a `Full` link next to the candidate name:

   ```text
   1. YATAI SMART INDUSTRIAL NEW CITY  Full
      Matched Name: MYANMAR YATAI INTERNATIONAL HOLDING GROUP CO., LTD.
      Record ID: NK-223CQDBzp8MRkdJMDiqXn3
      Score: 0.92 (similar-name)
   ```

4. User taps `Full`.
5. Telegram opens the bot deep link and sends `/start full_NK-223CQDBzp8MRkdJMDiqXn3`.
6. Bot checks whether the user is authorized.
7. If authorized, bot returns the same output as `/full <complete name>` for that record.
8. If unauthorized, bot returns the existing unauthorized response.

The main path requires one user action after seeing search results: tap `Full`.

## Chosen Approach

Render a row-level `Full` text link for each fuzzy candidate and handle `/start full_<recordId>` as a full-detail lookup.

The formatter should no longer return fuzzy candidate buttons. Instead, fuzzy candidate text should contain link markup for each candidate. The bot's Telegram adapter should send the fuzzy result with the parse mode or message entities needed to render the `Full` link.

Recommended displayed text:

```text
Possible matches

These are fuzzy name candidates, not a Debarred verdict. Tap Full to view sanctions details for a candidate.

1. YATAI SMART INDUSTRIAL NEW CITY  Full
   Matched Name: MYANMAR YATAI INTERNATIONAL HOLDING GROUP CO., LTD.
   Record ID: NK-223CQDBzp8MRkdJMDiqXn3
   Score: 0.92 (similar-name)
```

The word `Full` should be the only clickable text in each candidate row. Do not include `/basic` links in fuzzy search results.

## Alternatives Considered

### Keep inline callback buttons

This gives the most direct bot callback behavior, but it leaves the existing problem: a long button block appears under the result list.

### Use `/full <complete name>` command text per row

Telegram highlights slash commands, but command text with long names is awkward. Depending on client behavior, users may still need to copy, edit, or send extra text. This does not satisfy the one-tap target reliably.

### Use row-level deep links

This best matches the requested experience: links live next to the relevant candidate, and one tap can trigger the full lookup. It requires `/start` payload handling and bot username configuration, but avoids adding a large inline keyboard.

## Component Changes

### Configuration

Add a way for the formatter or bot adapter to know the bot username used in deep links.

Preferred source: derive it from configuration, such as a `TELEGRAM_BOT_USERNAME` environment variable, because the token does not safely or directly provide the public username at formatting time.

If the username is missing, the bot should avoid rendering invalid deep links. The safest fallback is to render plain candidate rows with `Record ID` and no `Full` link, and include a short operational warning in logs. The user-facing search result should remain readable.

### Domain Types

Extend the bot reply model to support Telegram link rendering without forcing every reply to use parse mode.

One acceptable shape:

```ts
interface BotReply {
  text: string;
  buttons: ReplyButton[][];
  notifications?: BotNotification[];
  parseMode?: 'HTML';
}
```

HTML is preferred because escaping candidate names and URLs is straightforward and avoids Markdown edge cases in sanction names.

### Formatter

Update `formatFuzzySearchResult` so fuzzy candidates:

- Return no buttons.
- Include a `Full` link for each candidate when a bot username is available.
- Keep fuzzy candidate warning language and avoid any `Debarred` verdict wording.
- Keep candidate number, primary name, matched name, record id, score, and match reason.
- Escape candidate text when HTML parse mode is used.

Exact check/basic/full formatters should not be changed except for any shared reply type adjustments.

### Start Payload Handling

Update `/start` handling so it accepts an optional payload:

```text
/start full_<recordId>
```

If the payload starts with `full_`, parse the record id and run `fullByRecordId`.

Access control must happen before lookup. Unauthorized users should get the existing unauthorized path instead of full details.

If the payload is malformed or the record id is empty, return the normal start/help response rather than throwing.

### Telegram Adapter

Update `replyToContext` to pass parse mode when present:

```ts
ctx.reply(reply.text, {
  parse_mode: reply.parseMode,
  reply_markup: ...
})
```

If a reply has no buttons, it should still be able to send parse mode for row-level links.

## Error Handling

- No fuzzy candidates: keep `No close name candidates found. Try a more complete name.` with no links and no buttons.
- Missing bot username: return readable fuzzy result text without invalid links.
- Unauthorized deep-link click: return existing unauthorized response.
- Malformed `/start full_...` payload: return normal start response or `Invalid action.`; prefer normal start response to avoid confusing first-time users.
- Unknown record id: preserve existing `fullByRecordId` behavior and return `No Data Found!`.
- Message truncation: links are generated only for candidates included in the formatted visible result.

## Testing Plan

Add or update Vitest coverage for:

- Fuzzy formatter returns no candidate buttons.
- Fuzzy formatter includes one `Full` link per candidate when bot username is configured.
- Link payload uses `full_<recordId>`.
- Fuzzy miss returns no buttons and no full links.
- Plain text fuzzy search uses the same row-level full-link behavior.
- `/search` fuzzy search uses the same row-level full-link behavior.
- `/start full_<recordId>` returns full sanctions details for an authorized user.
- `/start full_<recordId>` rejects unauthorized users.
- Malformed full deep-link payload does not crash.
- Existing exact check results still keep their current inline `/basic` and `/full` buttons.
- Existing callback behavior for exact-result buttons still works.
- Fuzzy search still does not start with `Debarred`.

## Out of Scope

Do not add:

- Row-level `Basic` links for fuzzy candidates.
- A new `/detail` command.
- A new `detail:<recordId>` callback action.
- A Telegram mini app or external web page.
- Changes to fuzzy ranking, candidate caps, exact matching, data refresh, or access control policy.
